<h2 id="repo-instructions">1 · Repository-Wide Instructions</h2>

<div class="card card-accent-blue">

**File:** `.github/copilot-instructions.md`

**Format:** Plain Markdown — use headings and subheadings to organize standards.

This file is **auto-attached to every Copilot interaction** in the repository. Think of it as the project's constitution — every prompt, completion, and agent session reads it.

</div>

<div class="code-block">
<div class="code-header">.github/copilot-instructions.md</div>

```markdown
# Project Standards

## Architecture
- Use repository pattern for data access
- Controllers never access the database directly

## Testing
- All source changes require corresponding tests
- Use Jest for unit tests, Playwright for E2E

## Style
- TypeScript strict mode, no `any` types
- Prefer functional components with hooks
```

</div>

<h2 id="path-based">2 · Path-Based Instructions</h2>

<div class="card card-accent-green">

**Directory:** `.github/instructions/`

**Files:** `*.instructions.md` with YAML `applyTo` frontmatter

Path-based instruction files fire **only when Copilot touches files matching the glob pattern**. They layer on top of the global instructions, so you can give language-specific or domain-specific guidance without cluttering the root file.

</div>

<div class="code-block">
<div class="code-header">.github/instructions/typescript.instructions.md</div>

```markdown
---
applyTo: "**/*.ts,**/*.tsx"
---
# TypeScript Standards
Use strict TypeScript. Never use `any`.
Prefer interfaces over types for object shapes.
All exported functions need JSDoc comments.
```

</div>

<div class="code-block">
<div class="code-header">.github/instructions/python.instructions.md</div>

```markdown
---
applyTo: "**/*.py"
---
# Python Standards
Use type hints on all function signatures.
Prefer dataclasses over plain dicts.
Use pytest for testing.
```

</div>

### Directory Structure

<div class="code-block">
<div class="code-header">Layout</div>

```
.github/
  copilot-instructions.md           # Global
  instructions/
    typescript.instructions.md      # TS/TSX files
    python.instructions.md          # Python files
    security.instructions.md        # Security-sensitive paths
    api.instructions.md             # API routes
```

</div>

### Glob Pattern Examples

<div class="card-grid">
  <div class="card">
    <code>**/*.py</code> — all Python files
  </div>
  <div class="card">
    <code>src/components/**/*.tsx</code> — React components
  </div>
  <div class="card">
    <code>**/*.test.ts</code> — test files only
  </div>
  <div class="card">
    <code>**/subdir/**/*.py</code> — Python files in any <code>subdir</code>
  </div>
</div>

<h2 id="skills">3 · Skills</h2>

<div class="card card-accent-purple">

**Project skills:** `.github/skills/<skill-name>/SKILL.md`

**Personal skills:** `~/.copilot/skills/<skill-name>/SKILL.md`

When Copilot invokes a skill, the **entire `SKILL.md` plus any supplementary files** in the skill directory are injected into context. You can include scripts, examples, and templates alongside the manifest.

</div>

<div class="code-block">
<div class="code-header">.github/skills/deploy/SKILL.md</div>

```markdown
---
name: deploy
description: Deploy the current project to production infrastructure
license: MIT
---
# Deploy Skill

## Steps
1. Run `npm run build` to create production bundle
2. Run `npm run test` to verify no regressions
3. Use `az webapp deploy` with the output from step 1
4. Verify deployment health with `curl https://...`

## Rules
- Never deploy without passing tests
- Always create a git tag after successful deploy
```

</div>

<h2 id="user-level">4 · User-Level Instructions</h2>

<div class="card card-accent-blue">

**Windows:** `%USERPROFILE%/copilot-instructions.md`

**macOS / Linux:** `~/copilot-instructions.md`

Personal preferences that apply **across every repository** you work in. Great for OS defaults, response style, and workflow habits.

</div>

<div class="code-block">
<div class="code-header">~/copilot-instructions.md</div>

```markdown
# My Preferences
- Always use PowerShell syntax for shell commands (I'm on Windows)
- Prefer concise responses with code examples
- When creating git commits, use conventional commit format
```

</div>
